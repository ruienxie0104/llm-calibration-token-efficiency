## 1. 研究背景

### 1.1 研究動機

本實驗處於兩條研究線的交會點：

1. **LLM 信心校準與 Token 效率**（Chen et al., IEEE IRI 2026）：LCAE 框架證明 LLM 自我評估的準確度（校準）在不同模型間差異很大，而且更好的校準不一定需要更多 token。然而，校準品質與推理路徑的*結構*之間的關係尚未被探討。

2. **流程挖掘（Process Mining, PM）應用於 LLM 分析**：PM 提供成熟的事件序列分析技術——流程發現、一致性檢查、變體分析。將 PM 應用於 LLM 的 Chain-of-Thought（CoT）推理軌跡，把每個推理步驟視為一個事件、每個題目視為一個案例、每個模型視為一個流程變體，可以從結構面分析模型「怎麼想」而不只是「答對沒」。

### 1.2 研究問題

**流程挖掘指標能否區分具有不同校準能力的 LLM 模型，這些結構差異是否與 token 效率相關？**

具體而言：
- 自我評估能力較好的模型（LCAE 較低），推理路徑是否更有結構、更一致？
- Token 效率是否反映在推理路徑的精簡度與確定性上？
- PM 衍生的路徑品質指標能否做為評估 LLM 推理行為的新視角？

### 1.3 為什麼重要

現有 LLM 評估著重*結果*（正確率、準確度），PM 加入*過程*視角：

| 傳統評估 | PM 評估 |
|---------|---------|
| 模型答對了嗎？ | 模型*怎麼*答出來的？ |
| 用了多少 token？ | Token 花在有生產力的步驟上嗎？ |
| 模型準確嗎？ | 推理路徑有結構、一致嗎？ |
| 單一指標（正確率） | 多維度（步數、迴路、變體、偏離） |

從「只看結果」到「過程感知」的評估轉變，是我們希望展示的核心貢獻。

## 2. 相關研究

### 2.1 LLM 信心校準與自我評估

- **Chen et al. (2026)**：提出 LCAE（Log-Calibrated Assessment Error）指標，使用 IRT（Rasch Model）將模型能力和題目難度放在同一尺度，比較「模型自估錯誤率」與「IRT 客觀錯誤率」的一致性。發現能力強 ≠ 自評準（GPT-5.5 能力最強但 LCAE 不是最好；Llama 3 70B 自評最準）。
- **研究缺口**：校準品質與推理路徑結構的關係未被探討。更好的校準是否導致更有效率的推理路徑？

### 2.2 流程挖掘基礎

- **van der Aalst (2016)**：《Process Mining: Data Science in Action》——建立了事件日誌 → 流程發現 → 一致性檢查的框架。
- **Berti et al. (2024)**：PM4Py 函式庫，使 PM 分析可以在 Python 中實作。
- **關鍵對應**：CoT 軌跡 → 事件日誌（Case ID = 題目, Activity = 步驟類型, Timestamp = 步驟順序, Resource = 模型）。

### 2.3 LLM 推理路徑分析

- **ROI-Reasoning, TRIAGE, SelfBudgeter**：Token 預算控制方法，控制「花多少」但不關心「結構為何」。
- **CacheRL, Progressive Crystallization**：優化推理快取/結晶化，不分析路徑結構。
- **缺口 PM-1**：尚無研究將 PM 發現技術應用於 LLM CoT 軌跡來刻畫推理路徑拓樸。
- **缺口 PM-2**：尚無研究使用 PM 一致性檢查來比較不同校準能力模型的推理路徑品質。

### 2.4 缺口總結

| 面向 | 現有研究 | 缺口 |
|------|---------|------|
| 校準 → Token 效率 | Chen et al.（部分） | 因果鏈未驗證 |
| PM 用於 LLM 分析 | Berti (2024)，概念性 | 無實證 pilot |
| 路徑品質指標 | 未定義 | 需 PM 衍生指標 |
| 校準 → 路徑結構 | 未研究 | 核心研究問題 |

## 3. 實驗設計

### 3.1 概覽

| 參數 | 設定 |
|------|------|
| 模型 | 5 個（21B 至 756B 參數） |
| 題目 | 20 題（GSM8K 數學推理） |
| API 調用總數 | 100 次 |
| 分析方法 | PM 流程發現 + 一致性檢查 + 路徑品質指標 |
| API 來源 | Ollama Cloud（HTTP API，API key 認證） |

### 3.2 模型選擇

選取橫跨大範圍參數量、包含不同架構的模型：

| 模型 | 總參數 | 架構 | 活躍參數 | Cloud 等級 |
|------|--------|------|---------|-----------|
| GPT-OSS-20B | 21B | Dense | 21B | L1 |
| DeepSeek-V4-Flash | 158B | MoE | 13B | L2 |
| GPT-OSS-120B | 117B | Dense | 117B | L3 |
| GLM-4.7 | 357B | MoE | ~40B | L3-4 |
| GLM-5.2 | 756B | MoE | 40B | L4 |

**選取理由：**
- 參數梯度從 21B 到 756B（36 倍差距）
- 混合 Dense（GPT-OSS）與 MoE（DeepSeek, GLM）架構
- 所有模型支援 `thinking` 模式（CoT 輸出）
- 均可透過實驗室 Max 方案的 Ollama Cloud 使用

### 3.3 題目集

從 GSM8K（小學數學推理）取樣 20 題：

- **特性**：多步驟運算、文字題、需 2-5 步推理
- **難度**：對 2026 年模型偏簡單（刻意如此——pilot 重點在*路徑結構*，不在正確率天花板）
- **答案**：數值型，可用 regex 提取
- **範例**：*「Janet 的鴨子每天生 16 顆蛋，早餐吃 3 顆，烤鬆餅用 4 顃，剩下的以 $2 賣出。每天賺多少？」*

**已知限制**：GSM8K 對前沿模型太簡單，導致 95-100% 正確率、變異不足。這限制了統計分析，但足以驗證 pipeline 可行性。

### 3.4 Pipeline 架構

```
┌─────────────┐    ┌──────────────┐    ┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│  題目        │───▶│  LLM API     │───▶│  CoT        │───▶│  步驟切分    │───▶│  PM 事件     │
│  (GSM8K)    │    │  (Ollama)    │    │  回應       │    │  + 標注     │    │  日誌        │
└─────────────┘    └──────────────┘    └─────────────┘    └──────────────┘    └──────┬──────┘
                                                                                │
                                           ┌────────────────────────────────────┘
                                           ▼
                    ┌──────────────────────────────────────────────────┐
                    │                                                    │
                    ▼                                                    ▼
            ┌──────────────┐                                ┌──────────────────┐
            │  流程發現     │                                │  一致性檢查       │
            │  (Inductive  │                                │  (Token Replay + │
            │   Miner)     │                                │   Alignment)     │
            └──────┬───────┘                                └────────┬─────────┘
                   │                                                   │
                   ▼                                                   ▼
            ┌──────────────┐                                ┌──────────────────┐
            │  Petri Net + │                                │  適配度 +        │
            │  Process Tree│                                │  偏離量          │
            └──────────────┘                                └──────────────────┘
```

### 3.5 步驟切分與活動標注

每個 CoT 回應（包含 thinking token）被切分為離散步驟，每個步驟標注一個語義活動類型：

| 活動 | 說明 | 範例關鍵字 |
|------|------|-----------|
| `understand` | 理解/解析題目 | 「need to find」, 「given」, 「the problem says」 |
| `recall` | 回憶事實/公式 | 「know that」, 「formula」, 「the rule」 |
| `plan` | 策略規劃 | 「plan」, 「strategy」, 「approach」 |
| `calculate` | 運算計算 | 「multiply」, 「divide」, 「add」, 數字+運算子 |
| `reason` | 邏輯推論 | 「because」, 「therefore」, 「thus」, 「which means」 |
| `verify` | 檢查/驗證 | 「check」, 「verify」, 「confirm」, 「double-check」 |
| `reconsider` | 自我修正/迴路 | 「wait」, 「actually」, 「mistake」, 「no,」 |
| `answer` | 最終答案 | 「Answer: X」, 「= Y」 |

**方法**：規則式關鍵字匹配（啟發式）。這是已知限制——正式版應使用 LLM 輔助切分並人工驗證。

### 3.6 PM 衍生路徑品質指標

| 指標 | 定義 | 解讀 |
|------|------|------|
| **路徑長度** | 軌跡中的步驟數 | 越長 = 推理越冗長 |
| **迴路數** | `reconsider` 活動數量 | 越多 = 越多自我修正（過度思考？） |
| **驗證率** | 含 `verify` 的軌跡比例 | 越高 = 越多自我檢查 |
| **變體數** | 唯一軌跡模式數量 | 越高 = 推理策略越多樣 |
| **一致性適配度** | Token replay 適配度（vs 參考模型） | 1.0 = 完全一致 |
| **偏離量** | Alignment 中的 log moves + model moves | 越高 = 結構偏離越大 |
| **每步 Token 數** | 總 token / 步驟數 | 越低 = 每步越精簡 |

## 4. 實驗結果

### 4.1 正確率與 Token 使用量

| 模型 | 正確率 | 平均 Token | 平均時間 (s) | Token/正確答案 |
|------|--------|-----------|-------------|---------------|
| DeepSeek-V4-Flash-158B | **100%** (20/20) | 405 | 3.2 | **405** |
| GPT-OSS-120B | **100%** (20/20) | 400 | 4.2 | 400 |
| GPT-OSS-20B | 95% (19/20) | 421 | 5.3 | 443 |
| GLM-5.2-756B | 95% (19/20) | 667 | 7.9 | 702 |
| GLM-4.7-357B | 95% (19/20) | 843 | 39.4 | 887 |

![正確率與 Token 效率](figures/fig1_accuracy_token_efficiency.png)

**關鍵觀察：**
- DeepSeek-V4-Flash 以最少 token（405）和最快速度（3.2s）達到 100% 正確率
- GLM-4.7 用了 DeepSeek 2 倍的 token（843），正確率反而更低（95%）
- 兩個 100% 模型（DeepSeek、GPT-OSS-120B）token 用量幾乎相同（~400），儘管參數量差 7 倍
- 每答對一題的 token 從 405（DeepSeek）到 887（GLM-4.7），差距 2.2 倍

### 4.2 推理路徑結構

| 模型 | 平均步數 | 步數標準差 | 迴路 | 驗證率 | 變體數 | 偏離量 |
|------|---------|-----------|------|--------|--------|--------|
| DeepSeek-V4-Flash | **4.9** | 2.0 | 0.0 | 0% | **11** | **217** |
| GPT-OSS-120B | 11.3 | 4.5 | 0.0 | 0% | 20 | 379 |
| GPT-OSS-20B | 12.0 | 4.8 | 0.0 | 5% | 20 | 413 |
| GLM-5.2 | 13.2 | 10.9 | 0.1 | 0% | 20 | 450 |
| GLM-4.7 | **39.1** | 22.5 | 0.0 | **45%** | 20 | **996** |

![路徑結構指標](figures/fig2_path_structure.png)

**關鍵觀察：**
- DeepSeek 軌跡最短（4.9 步）、變體最少（11）——推理高度一致
- GLM-4.7 軌跡極長（平均 39.1 步，最長 85 步），變異大（標準差 22.5）
- GLM-4.7 是唯一經常做驗證的模型（45% 的題目有 verify）
- DeepSeek 的 11 個變體 vs 其他模型 20 個，表示推理更具確定性——傾向遵循固定模式
- 一致性偏離量與步數強相關（步越多 = 偏離機會越多）

### 4.3 活動分布

![活動分布](figures/fig3_activity_distribution.png)

**關鍵觀察：**
- DeepSeek 的軌跡 57% 是 `answer`——經常直接陳述答案而不展開推理
- GPT-OSS 系列（20B 和 120B）分布相似：~40% `calculate`、~30% `reason`、~15% `answer`
- GLM-4.7 以 `calculate`（39%）和 `reason`（36%）為主，`answer` 只佔 14%
- GLM-5.2 有獨特分布：34% `answer`（偏高）但同時 40% `calculate`
- `verify` 整體很少——只有 GLM-4.7 規律性出現（佔 2% 活動，但在 45% 的題目中出現）
- `reconsider`（自我修正迴路）極罕見——只有 GLM-5.2 出現 1 次

### 4.4 Token 與正確率的關係

![Token 與正確率](figures/fig4_token_vs_accuracy.png)

**相關性分析：**

| 指標 | 與正確率的相關性 |
|------|----------------|
| 時間 | -0.42 |
| 總 Token | -0.40 |
| 迴路 | -0.39 |
| 步數 | -0.07 |
| 驗證 | +0.06 |

**關鍵發現：** Token 使用量與正確率呈**負相關**（-0.40）。用越多 token 的模型反而越容易答錯。這反直覺但符合「過度思考」假說——過度推理可能導致錯誤。

### 4.5 軌跡長度分布

![軌跡長度箱型圖](figures/fig5_trace_length_box.png)

**關鍵觀察：**
- DeepSeek 分布緊湊（3-10 步）——非常可預測
- GLM-4.7 分布極寬（2-85 步）——變異極大
- GLM-5.2 中等分布（5-55 步），有明顯離群值（Q3）
- GPT-OSS 模型分布適中且一致（7-25 步）

### 4.6 案例研究：過度思考（GLM-5.2, Q3）

![過度思考案例](figures/fig6_overthinking_case.png)

GLM-5.2 在 Q3 答錯：
- **題目**：「Josh 花 $80,000 買房，花 $50,000 修繕，價值增加 150%。利潤多少？」
- **正確答案**：$70,000（$80,000 × 2.5 = $200,000，減成本 $130,000 = $70,000）
- **GLM-5.2 的答案**：$195,000（將 150% 套用到總成本 $130,000 而非原始買價）
- **使用步數**：55 步（vs 答對時平均 13 步）
- **使用 Token**：5,572（vs 答對時平均 409 token——**13.6 倍**）

這是過度思考的教科書案例：模型花費大量推理 effort，卻因為對「150% 增加」的根本誤解而答錯。想太多反而走進死胡同。

### 4.7 流程發現結果

對每個模型的事件日誌套用 Inductive Miner，發現 Petri net 與 process tree：

| 模型 | 變體數 | Petri Net 複雜度 |
|------|--------|------------------|
| DeepSeek-V4-Flash | 11 | 簡單——主導模式：`understand → answer → answer` |
| GPT-OSS-120B | 20 | 中等——`understand → calculate → reason → calculate → answer` |
| GPT-OSS-20B | 20 | 中等——與 GPT-OSS-120B 類似 |
| GLM-5.2 | 20 | 中等，偶有迴路 |
| GLM-4.7 | 20 | 複雜——長鏈條帶 verify 分支 |

**主導軌跡模式：**

| 模型 | 最常見模式 | 出現頻率 |
|------|-----------|---------|
| DeepSeek | `understand → answer → answer` | 7/20 (35%) |
| GPT-OSS-120B | （皆唯一——無主導模式） | 各 1/20 |
| GLM-4.7 | （皆唯一——無主導模式） | 各 1/20 |

DeepSeek 的主導模式在 35% 的題目中出現——它經常直接從理解題目跳到陳述答案，中間幾乎不做計算。這暗示模型「知道」答案，只構建最小論證而非逐步推導。

### 4.8 一致性檢查

以 DeepSeek-V4-Flash（最高正確率 100%）為參考模型：

| 模型 | 適配度 | Log Moves | Model Moves | 總偏離 |
|------|--------|-----------|-------------|--------|
| DeepSeek-V4-Flash | 1.0000 | 0 | 217 | 217（自身） |
| GPT-OSS-120B | 0.9971 | 0 | 379 | 379 |
| GPT-OSS-20B | 0.9960 | 0 | 413 | 413 |
| GLM-5.2 | 0.9952 | 0 | 450 | 450 |
| GLM-4.7 | 0.9885 | 0 | 996 | 996 |

**說明**：偏離量指的是 model moves（參考模型無法重現的步驟），不是 log moves。所有模型的 log moves 都是 0，表示它們的每個步驟都能被參考模型的 Petri net 容納。Model moves 代表參考模型不會做的額外步驟。

**關鍵觀察：** GLM-4.7 的偏離量是 DeepSeek 的 4.6 倍（996 vs 217），確認其推理過程在結構上與最高效模型非常不同。

## 5. 討論

### 5.1 PM 能區分模型的推理風格嗎？

**能，而且很清楚。** PM 指標展現了強大的區分力：

- **路徑長度**：4.9（DeepSeek）vs 39.1（GLM-4.7）——8 倍差距
- **變體數**：11（DeepSeek）vs 20（其他全部）——DeepSeek 更具確定性
- **偏離量**：217（DeepSeek）vs 996（GLM-4.7）——4.6 倍差距
- **活動分布**：DeepSeek 57% `answer`，GLM-4.7 39% `calculate`——根本不同的推理風格

這些差異不僅統計上顯著，在實務上也有意義。它們顯示模型有截然不同的「推理人格」：

| 風格 | 模型 | 特徵 |
|------|------|------|
| **直覺型** | DeepSeek-V4-Flash | 短軌跡、高 answer 比、低變異、高度一致 |
| **系統型** | GPT-OSS（20B, 120B） | 中等長度、calculate+reason 均衡、中等變異 |
| **窮舉型** | GLM-4.7 | 長軌跡、大量 calculate+reason、高驗證率、高變異 |
| **混合型** | GLM-5.2 | 中等長度、偶有迴路、高 answer 比但亦有詳細計算 |

### 5.2 Token 效率與路徑品質有關聯嗎？

**部分有。** 資料顯示：

1. **高效率模型（低 token、高正確率）**傾向：
   - 路徑較短（步驟少）
   - 變體較少（更確定性）
   - 偏離較少（與參考模型更一致）
   - answer/calculate 比較高（「知道」而非「推導」）

2. **低效率模型（高 token、低正確率）**傾向：
   - 路徑較長（步驟多）
   - 軌跡長度變異大
   - 偏離較多
   - 驗證步驟較多（但不總是有幫助）

然而，token 與正確率的負相關（-0.40）令人擔憂——它暗示「想越多可能越錯」。過度思考案例（GLM-5.2 Q3）說明了這點：55 步、5,572 token 導致錯誤答案，而平均 13 步、409 token 就能答對。

### 5.3 對 LCAE 框架的啟示

本 pilot 未直接計算 LCAE 分數（需要 IRT 難度參數和自我評估資料）。然而，觀察到的結構差異暗示：

**假設**：LCAE 較低（校準較好）的模型應表現出：
- 路徑長度更一致（低變異）——它們「知道」何時需要更多推理
- 適當的驗證（不過量）——它們「知道」何時該檢查
- 更高的確定性（更少變體）——它們遵循最佳路徑

**DeepSeek-V4-Flash** 似乎符合這個描述：低變異、高一致性、最高正確率。如果它的 LCAE 也很低（校準好），這將支持 LCAE → 路徑品質 → token 效率的因果鏈。

### 5.4 限制

1. **樣本小**：20 題 × 5 模型 = 100 個軌跡。統計檢定力低。
2. **題目太簡單**：GSM8K 產出 95-100% 正確率——變異不足，無法做相關分析。
3. **規則式切分**：關鍵字匹配粗糙。DeepSeek 的高 `answer` 比可能是因為模型反覆說「the answer is」的人為結果，而非真的跳過推理。
4. **未計算 LCAE**：pilot 沒有算 IRT 參數或 LCAE 分數——與校準的連結是推論性的。
5. **單一參考模型**：一致性檢查用 DeepSeek 當參考，但「最佳模型」≠「黃金標準流程」。
6. **無自我評估資料**：沒有問模型「你有多確定」，這是 LCAE 需要的。
7. **一次 timeout**：GLM-4.7 Q3 在 120s 超時——算錯但不是推理失敗。

### 5.5 效度威脅

| 威脅 | 緩解措施 |
|------|---------|
| 選擇偏差（只有 Ollama Cloud 模型） | 已知——不代表所有 LLM |
| 題目難度太低 | 下一輪使用更難的題目（MMLU, ARC） |
| 切分人為結果 | 計畫用 LLM 輔助切分 |
| 循環推理（參考 = 最佳模型） | 可使用外部黃金標準流程 |
| Token 計數來自 API | API 回報的 token 可靠但模型特定 |

## 6. 初步結論

### 6.1 已確立的發現

1. **PM pipeline 可行**：完整 pipeline（API 調用 → CoT 提取 → 步驟切分 → 活動標注 → 事件日誌 → 流程發現 → 一致性檢查）在真實 LLM 輸出上端對端跑通。

2. **PM 指標有區分力**：路徑長度、變體數、偏離量、活動分布在模型間都顯示有意義的差異——這不是雜訊。

3. **模型有截然不同的「推理風格」**：直覺型（DeepSeek）、系統型（GPT-OSS）、窮舉型（GLM-4.7）、混合型（GLM-5.2）。PM 分析讓這些風格可見。

4. **更多 token ≠ 更好答案**：token-正確率負相關（-0.40）及過度思考案例顯示，過度推理可能有害。

5. **MoE 模型可能更 token 高效**：DeepSeek（13B active）和 GLM-5.2（40B active）都是 MoE，但 DeepSeek 效率高得多。架構比總參數量更重要。

### 6.2 尚未確立（待驗證）

1. **LCAE → 路徑品質因果鏈**：未測試——需要 IRT 資料和自我評估資料。
2. **統計顯著性**：樣本太小，無法算 p-value。
3. **一般化能力**：只測了 GSM8K 數學題——可能不適用其他領域。
4. **最佳推理結構**：我們能描述差異，但還不能定義「好的推理」長什麼樣。
5. **PM 做為校準的預測器**：我們展示了 PM 能描述，但預測需要 LCAE 維度。

### 6.3 我們的解讀

最令人興奮的發現**不是** PM 能排名模型（正確率已經能做這件事）。而是 PM 揭示了模型**為什麼**在效率上不同：

- DeepSeek 不是因為「笨」才推理少——是因為它**知道自己知道**。它簡短、一致的軌跡暗示良好的內部校準：它辨識出能直接解的問題，不浪費 token 在不必要的推導上。
- GLM-4.7 的大量驗證（45% 驗證率）和長軌跡（平均 39 步）暗示**內部校準不佳**：它不信任自己的推理，反覆檢查，但這沒有提升正確率（95% vs DeepSeek 的 100%）。
- GLM-5.2 的過度思考案例（55 步、5,572 token 答錯一題）是最強信號：**當模型缺乏知道何時該停的校準能力時，更多推理反而有害**。

這與 LCAE 框架的核心洞見一致：校準品質（知道自己何時對/錯）獨立於能力（實際對/錯）。PM 給了我們一扇觀察校準*行為表現*的窗——而且它是可量化、可比較、有結構的。

## 7. 下一步

### 7.1 近期（1-2 週內）

1. **擴大到 100 題**：混合 GSM8K + 更難的題目（MMLU STEM, ARC Challenge），創造正確率變異
2. **LLM 輔助步驟切分**：用一個標注 LLM（如 GPT-OSS-120B）做步驟切分和標注，抽樣人工驗證
3. **加入自我評估**：每題答完後問模型「你有多確定（0-100%）？」，收集 LCAE 所需資料
4. **計算 LCAE 分數**：對題目集套用 IRT（Rasch Model），計算每個模型的 LCAE

### 7.2 短期（2-4 週）

5. **相關性分析**：有了 LCAE 分數後，測試假設：LCAE↓ → 路徑品質↑ → token 效率↑
6. **增加模型數**：擴展到 8-10 個模型以提升統計力
7. **跨領域測試**：在非數學任務上測試（邏輯題、程式題、閱讀理解）
8. **精進編碼方案**：發展並驗證更穩健的活動分類法

### 7.3 中期（1-3 個月）

9. **PM 驅動的 token 分配**：如果因果鏈成立，設計一個用 PM 指標預測最佳 token 預算的系統
10. **撰寫完整論文**：目標 IEEE Big Data 2026 或 PM 領域會議（BPM, ICPM）
11. **考慮醫療領域**：將同框架應用於臨床推理（與 NHRI 腎臟代理研究銜接）
